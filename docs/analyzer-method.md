# Метод отбора «золота» (analyzer)

Спецификация того, как goldcut выбирает короткие клипы из длинного ролика.
Реализация — `src/goldcut/analyzer/segmenter.py`; этот документ — её источник правды.

## Принцип: две разные задачи

«Нарезка» и «отбор золота» — НЕ одно и то же. Наивный путь (порезать транскрипт на
равные чанки и засаммаризировать) даёт последовательную кашу. Мы делаем два слоя:

1. **Сегментация (LLM).** Найти отдельные самодостаточные мысли — границы по смыслу и
   предложениям, а не по таймеру.
2. **Скоринг (код).** Оценить каждую по рубрике, добавить heatmap-буст, взять top-K.
   Большая часть ролика обязана не дать ничего — это и есть «выуживание».

Разделение намеренное: веса и heatmap-математика живут в коде (прозрачно и тюнится),
LLM отвечает только за нахождение кусков и оценки по рубрике.

## Что делает LLM (editorial-бриф)

Роль — редактор коротких вертикальных видео. Вход — транскрипт с таймкодами `[MM:SS]`
плюс список «самых пересматриваемых» моментов (heatmap). Требования к каждому куску:

- **15–60 секунд**, границы по предложениям;
- **хук** в первые ~2 секунды, **развязка** в конце (не обрывается);
- **самодостаточность** — понятно без контекста ролика;
- **разнообразие** — разные мысли из разных мест, не вариации одной.

**Исключается** (детектится и выкидывается): реклама/спонсорские чтения, само-промо
канала (подписка, лайки, промокоды, ссылки), организационные подводки, повторы, оффтоп.

heatmap — **сигнал, а не диктат**: подсказывает, где зрители залипают, но LLM судит сам.

Выход (structured output, Pydantic `SegmentSelection`): 10–15 кандидатов, у каждого
`start_s`/`end_s` (секунды), `title`, `summary`, `hook`, `transcript`, `scores`, `why`.

## Рубрика и веса (скоринг в коде)

Каждый кандидат получает оценки 0–5 от LLM; код считает взвешенный балл:

| Критерий | Что оценивает | Вес |
|----------|---------------|-----|
| `self_contained` | понятно без контекста ролика | 0.25 |
| `insight` | неочевидность мысли | 0.25 |
| `hook` | сила первых ~3 секунд | 0.20 |
| `emotion` | эмоция / цитируемость | 0.15 |
| `payoff` | есть развязка, не обрывается | 0.15 |

**heatmap-буст:** макс. интенсивность heatmap (0..1), попадающая в `[start_s, end_s]`,
умноженная на `HEATMAP_SCALE` (0.5), прибавляется к взвешенному баллу.

`total = Σ(вес × оценка) + heatmap_boost`. Сортировка по `total` ↓, отдаём top-K.

Веса, `HEATMAP_SCALE`, `bucket_s` парсера и сам бриф — точки тюнинга по результатам прогонов.

## Привязка таймкодов (snap)

LLM возвращает верный текст клипа, но секунды иногда уезжают (бриф видит транскрипт,
сгруппированный по ~20с, и оценивает время на глаз). Поэтому после отбора код
**привязывает** каждый клип к точным секундам по пословному VTT (`transcript.parse_vtt_words`
даёт поток `(сек, слово)` из тегов `<00:00:02.000><c> at</c>`):

- `snap_timecodes` выравнивает текст клипа по потоку слов через `difflib`, **якорясь на
  самом длинном совпадающем блоке** и проецируя от него границы — это не даёт коротким
  ложным совпадениям (вроде «if there was a» в начале ролика) растянуть клип;
- привязанные секунды принимаются, только если длительность в пределах `SNAP_MIN_S..SNAP_MAX_S`
  (8–120с); иначе оставляем таймкоды LLM;
- heatmap-буст считается уже **после** привязки — от точных секунд.

Без привязки клип «как работает ИИ» уезжал на 65:36; после — корректные 40:10–40:47.

## Модель

`claude-opus-4-8` по умолчанию (тонкость отбора важнее цены; ~20k токенов транскрипта
влезают одним запросом). Настраивается через `GOLDCUT_ANALYZER_MODEL`. Вызов —
официальный Anthropic SDK, `client.messages.parse(... output_format=SegmentSelection)`
(валидированный structured output, без ручного парсинга JSON).

## Валидация подхода (2026-06-25)

Прогон вручную на 88-мин подкасте (Diary Of A CEO × Roman Yampolskiy): heatmap-пики
совпали с независимым отбором (главный пик ролика 53:27 = клип №1), реклама корректно
отсеяна, из 88 минут выбрано ~11 кусков по 30–60с в разных местах. Подтвердило, что
связка «LLM-редактор + heatmap-буст» находит золото, а не угадывает.
